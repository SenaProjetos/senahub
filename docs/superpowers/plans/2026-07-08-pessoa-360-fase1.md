## 1. Objetivo

1. **Ficha única "Pessoa 360" (somente leitura)**: uma rota que reúne, por pessoa, tudo o que hoje exige abrir 5 telas —
   com **abas condicionais ao vínculo** (CLT/estagiário veem ponto/folha/férias; PJ vê cadastro PF + a PJ vinculada;
   cliente vê os dados do cliente).
2. **Fechar o buraco das ausências**: hoje não existe visão **por pessoa** de abonos de falta e férias (só a fila global
   de pendências em `/rh/admin` e a self-service do próprio colaborador). A ficha ganha uma aba **Ausências**.
3. **Item 6a — aproveitar dados na criação**: ao aprovar uma `SolicitacaoCadastro`, abrir a criação de usuário
   **pré-preenchida** (nome/e-mail/telefone) em vez de redigitar tudo.

**Fora de escopo desta fase** (ficam para fases seguintes do parecer):
- **Edição** na ficha (a 360 aqui é read-only; escrita continua nas telas atuais). Auto-serviço com validação = Fase 4.
- Assistente unificado de criação completo (wizard acesso→vínculo→cadastro→escala) = Fase 2. Só o 6a (pré-preencher) entra aqui.
- **Log de leitura** de aba sensível no `AuditLog` = Fase 3. Aqui já entra o **gate por aba**, mas não o registro de leitura.
- Permissões por usuário/roles sob medida = Fase 5.

## 3. Arquitetura da página

Padrão **mestre-detalhe** com **abas condicionais**, tudo em Server Components:

- **Lista** (`/rh/pessoas`): reusa `listarUsuarios()` (já traz `role`, `ativo`, `clienteId`, sócio) + etiquetas
  (vínculo, "cadastro incompleto" derivado dos campos). Filtro/paginação via `parseListParams`.
- **Detalhe** (`/rh/pessoas/[id]`): **cabeçalho fixo** (resumo) + **abas carregadas sob demanda**.
  - Cada aba é um **Server Component próprio** que só busca seus dados quando aberta (via segmento paralelo/`Suspense`),
    para a ficha não disparar 10 queries de uma vez.
  - **Regra de ouro (do parecer):** cada aba **reusa** o componente/So query do módulo dono. A 360 **orquestra**, não
    reimplementa regra de negócio.

### Cabeçalho / resumo (topo)
Vínculo (`ROLE_LABELS`), status (`ativo` + presença online via `usuarioOnline`), admissão/valor
(`dataAdmissao`, `salarioBase` ou valor PJ), projetos ativos (`escopoProjeto`), e **pendências** (documento faltando,
ponto em aberto, ausência aguardando validação). Gate: campos sensíveis do resumo (salário) só com a permissão da aba Folha.

## 4. Matriz de abas × vínculo (o coração da fase)

Abas aparecem **só** quando cabem ao vínculo. Cada aba **reusa uma query já existente** e é **gateada** por permissão fina.

| Aba | CLT | Estag. | PJ (projetista/free) | Interno¹ | Cliente | Fonte reusada | Gate (permissão) |
|-----|:--:|:--:|:--:|:--:|:--:|----|----|
| **Resumo** (topo) | ✓ | ✓ | ✓ | ✓ | ✓ | agregação | sessão + escopo |
| **Acesso** (role, ativo, mustChange, sócio) | ✓ | ✓ | ✓ | ✓ | ✓ | `User` | `usuarios:gerir` |
| **Cadastro (PF)** | ✓ | ✓ | ✓ | ✓ | — | `listarFuncionarios()` → variante por `userId` (§6) | `rh:cadastro`² |
| **Contrato / PJ** | — | — | ✓ | — | — | `User.pj` + `listarPessoasJuridicas()` | `rh:cadastro` |
| **Ponto & Banco de horas** | ✓ | ✓ | — | (se bate ponto) | — | `bancoHorasDe(userId)` + `acumuladoAte(userId,ano,mes)` + `modules/ponto` | `rh:admin` |
| **Ausências** (abonos + férias) | ✓ | ✓ | — | (interno) | — | `minhasSolicitacoes(userId)` (generalizar §6) | `rh:admin` |
| **Folha** (holerites) | ✓ | ✓ | — | — | — | `holerites where userId` (reusa `folha/queries`) | **sensível**³ |
| **NF** (notas fiscais) | — | — | ✓ | — | — | NF por `userId` (`modules/rh/nf`) | `rh:admin` |
| **Escala / Expediente** | ✓ | ✓ | (opt) | ✓ | — | `escalaUsuarioGrade(userId)` c/ fallback `escalaRoleGrade(role)` | `rh:admin` |
| **Documentos** | ✓ | ✓ | ✓ | ✓ | — | `funcDocumentos` (já em `listarFuncionarios`) | `rh:cadastro` |
| **Dados do cliente** | — | — | — | — | ✓ | `Cliente` por `clienteId` (`modules/clientes`) | `clientes:ver` |

¹ Interno = `administrativo`/`supervisor`/`ti` (e `admin`). ² `rh:cadastro` = recurso a **criar** no catálogo (não
existe hoje; `/rh/funcionarios` é gateada só por `HR_ADMIN_ROLES` na página — §5). ³ Folha carrega salário/holerite —
gate mais estrito (sócio/DP), separado de quem só administra acesso.

Derivação do vínculo: `CLT_ROLES`, `PJ_ROLES`, `INTERNAL_ROLES`, `role === "cliente"`, presença de `pjId`/`clienteId`
(todos já em `lib/roles.ts`). **Nenhuma aba nova de dados** — todas mapeiam relações que já existem no `User`.

## 5. Permissões & segurança

- **Gate por aba** reusando o catálogo fino (`lib/permissions-catalog.ts`): abrir a ficha ≠ ver todas as abas.
  Quem administra acesso (`usuarios:gerir`) não herda **Folha**; ver cadastro (`rh:cadastro`) não herda **Ponto**.
- **Recurso `rh:cadastro`**: **não existe** no catálogo fino hoje (só há `ponto`; RH é gateado apenas por
  `HR_ADMIN_ROLES` no nível de página). **Criar** `rh` (`cadastro`/`admin`) em `permissions-catalog.ts` é
  **pré-requisito** do gate por aba — sem ele, não há como separar "ver acesso" de "ver cadastro/folha".
- **Separação DP × projeto** (ressalva LGPD do conselho): ver ficha de projeto **não** libera Folha/Ponto.
- **Escopo**: `acessoGlobal(u)` (global/sócio) vê qualquer pessoa; demais só via `HR_ADMIN_ROLES`; cliente só a si.
- **Log de leitura sensível**: fora desta fase (Fase 3), mas **não** introduzir nada que dificulte adicioná-lo depois
  (centralizar o carregamento das abas sensíveis num ponto único).

## 6. Ajustes de query (pequenos, sem migração)

Generalizações de leituras que hoje são "todos" ou "eu" para "um `userId`":

- **Cadastro por pessoa**: extrair de `listarFuncionarios()` uma `fichaCadastroFuncionario(userId)` (mesmos `select`,
  filtrada por id) — evita puxar a lista inteira para 1 ficha.
- **Ausências por pessoa**: `minhasSolicitacoes(userId)` **já** aceita `userId` — só passar a expor para admin/RH (hoje é
  chamada com o próprio id). Renomear conceitualmente para `solicitacoesDoUsuario(userId)` (mantendo compat) e gatear.
- **Folha por pessoa**: `holerites where { userId }` (o model `Holerite` tem relação com `User`); reusar o render de holerite.
- **NF por pessoa**: filtrar as notas por `userId` (a query de `/rh/admin` já lista NFs — extrair a variante por pessoa).
- **Escala**: `escalaUsuarioGrade(userId)` já devolve `{ temOverride, dias }`; quando `!temOverride`, mostrar
  `escalaRoleGrade(role)` como "herdado do perfil". Zero código novo de negócio.

Todas são leituras `server-only`. Nenhuma altera schema.

## 7. Item 6a — aproveitar dados na criação

**Hoje:** `/solicitar-cadastro` grava `SolicitacaoCadastro { nome, email, telefone, mensagem }`;
`avaliarSolicitacaoCadastro` **apenas muda o status** ("não cria o usuário automaticamente"); o admin depois
**redigita** nome+e-mail em `criarUsuario`.

**Mudança — D2 (a confirmar):**
- **D2-A (recomendado):** ao aprovar, **abrir o formulário de criação pré-preenchido** (nome/e-mail/telefone da
  solicitação) — mantém o passo de **revisão do admin** (ressalva "reaproveitar sim, confiar não") e a escolha do vínculo.
  Implementação: `avaliarSolicitacaoCadastro(aprovar:true)` retorna os dados; a `usuarios-view` abre o
  `criar-usuario-dialog` já preenchido (querystring ou estado). Sem criar user automático.
- **D2-B:** aprovar **cria** o usuário direto (com senha temporária). Mais rápido, menos controle — não recomendado
  para esta fase (perde a revisão e a definição de vínculo/role no ato).

**Regras de segurança (do conselho):**
- E-mail da solicitação é **não verificado** → tratar como não-confirmado; a senha temporária/`mustChangePassword`
  já prova a posse (fluxo atual de `criarUsuarioComCredencial`).
- Formulário público continua **magro** (sem CPF/banco) — enriquecimento de cadastro é **pós-login** (Fase 2/4).
- Manter a dedução de duplicidade por e-mail pendente que já existe em `solicitarCadastro`.

Sem migração: só fluxo de UI + retorno da action.

## 10. Verificação

- **Vínculos**: abrir a ficha de um CLT (vê Ponto/Folha/Ausências/Escala), um `projetista_pj` (vê Cadastro PF + PJ + NF,
  **não** vê Folha/Ponto), um `cliente` (vê só Dados do cliente + Acesso). Confirmar que abas fora do vínculo **não renderizam**.

## 11. Riscos / pontos abertos

- **D1 (rota/local)** e **D2 (comportamento do aprovar)** precisam de OK antes de codar.
- **Acoplamento de UI**: alguns miolos de tela estão colados à página do módulo — extrair sem quebrar a tela original

- **Cliente na lista**: decidir se `cliente` sequer aparece em `/rh/pessoas` (recomendado: **não**; ele só tem a própria ficha).
- **Log de leitura sensível** (Fase 3): manter o carregamento das abas sensíveis num ponto único para plugar o log depois.
- **Escrita**: reforçar que esta fase é **read-only**; qualquer edição continua nas telas atuais até a Fase 4.
